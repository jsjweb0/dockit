# DocKit - 한글 문서 양식 작성 도구

> 입력하면 정해진 문서 양식 형태로 즉시 확인되는 웹 문서 작성 도구

[데모 보기](https://dockit.jsjweb0.workers.dev/) · [GitHub](https://github.com/jsjweb0/dockit)

DocKit 예시 불러오기와 PDF 저장 흐름

![DocKit demo](./docs/readme/dockit-demo.gif)

---

## 프로젝트 목적

Word나 한글 문서는 문서 프로그램이 없으면 정해진 양식의 문서를 작성하기 어렵습니다.
DocKit은 별도의 문서 프로그램 없이 웹에서 한글 파일 양식의 문서를 작성하고, 입력 결과를 실제 출력 형태에 가깝게 실시간 확인할 수 있도록 만든 도구입니다.

입력 폼과 미리보기를 함께 제공해 작성 중인 내용이 실제 문서 화면에서 어떻게 보이는지 바로 확인할 수 있습니다.

**개발 기간**: 2026.04 ~ 진행 중 (1인 개발)

---

## 주요 기능

| 기능            | 설명                                                         |
| --------------- | ------------------------------------------------------------ |
| 실시간 미리보기 | 입력값을 이력서, 자기소개서, 경력기술서 미리보기에 즉시 반영 |
| 자동 저장       | localStorage 저장과 이탈 방지 안내로 작성 중 손실 방지       |
| PDF 저장        | 브라우저 인쇄와 `@media print`로 텍스트 선택 가능한 PDF 출력 |
| 입력값 검증     | 필수값, 연락처/이메일/URL 형식, 경력 기간 순서 오류 확인     |
| 최근 작성 목록  | 여러 문서를 저장하고 이어 작성할 수 있는 문서 메타 정보 관리 |

### 문서별 기능

- **이력서**: 기본 정보, 학력, 자격증, 경력, 프로젝트, 링크, 기술 스택 입력
- **자기소개서**: 문항별 작성, 글자 수 확인, 실시간 미리보기
- **경력기술서**: 경력 추가/삭제, 업무 내용, 성과, 기술 스택 정리

---

## 기술 스택

- **React 19 + TypeScript**: 반복 섹션 추가/삭제와 실시간 미리보기 상태를 컴포넌트 단위로 관리
- **Context API**: 문서별 에디터 상태를 라우트 내부 Provider로 제한해 별도 상태관리 라이브러리 없이 구현
- **Radix UI**: AlertDialog, Tabs, Tooltip처럼 키보드 접근성이 중요한 UI 구성
- **Vite + Cloudflare Workers**: 빠른 빌드와 정적 배포, SPA 라우팅 지원
- **localStorage**: 백엔드 없이 문서 저장 흐름을 검증하고, `meta.version`으로 추후 데이터 변환 대비

## 어려웠던 점과 해결 방법

**초기 폴더 구조와 아키텍처 설계**  
처음에는 이력서 작성 기능만 고려해 구조가 단순했지만, 자기소개서와 경력기술서가 추가되면서 문서별 코드가 늘어났습니다. `features` 기준으로 문서 도메인을 나누고, `documents`에는 저장, 내보내기, 공통 검증 훅처럼 여러 문서가 공유하는 흐름을 모아 수정 범위를 줄였습니다.

**실시간 미리보기 상태 구조**  
입력 폼 데이터와 출력용 데이터를 따로 관리하면 동기화 버그를 추적하기 어렵습니다. 이력서, 자기소개서, 경력기술서 각각의 전체 문서 데이터를 하나의 객체로 관리하고, 폼과 미리보기가 같은 데이터를 바라보도록 설계했습니다.

**양식 확장을 고려한 에디터 구조**  
처음에는 이력서 1개 양식에서 시작했지만 자기소개서와 경력기술서가 추가되면서 공통 작성 흐름과 문서별 차이를 분리했습니다. `editor.config.ts`에서 Provider, 샘플 데이터, 최근 문서 저장소, 제목 생성 함수를 연결하고, 공통 `EditorLayout`은 문서 내부 구조를 몰라도 동작하도록 정리했습니다.

**Context 책임 축소**  
Context가 문서 상태, 검증, 템플릿 정보까지 함께 담당하면서 책임이 커졌습니다. Context에는 문서 상태와 저장/출력 흐름을 남기고, 템플릿 정보는 `documentTemplates.ts`, 검증 흐름은 `useDocumentValidation`과 문서별 validation hook으로 분리했습니다.

**검증 로직 분리**  
검증 로직은 UI 코드와 분리한 순수 함수로 작성했습니다. 문서마다 에러 구조는 다르지만 공통 훅이 사용하는 결과 형태를 통일해 저장/출력 전 검증 흐름을 재사용할 수 있게 했습니다.

**PDF 저장 방식**  
canvas 이미지 캡처 대신 브라우저 인쇄(`window.print`)와 `@media print`를 사용했습니다. 덕분에 저장된 PDF에서도 텍스트 선택이 가능합니다.

**표 형태 문서의 출력 품질**  
한글 파일 양식처럼 표 구조가 중요한 화면은 컴포넌트 분리만으로는 부족했습니다. 인쇄 스타일, 셀 너비, 긴 텍스트 줄바꿈까지 함께 조정해 화면 미리보기와 PDF 출력 형태가 크게 어긋나지 않도록 개선했습니다.

**테스트 범위 분리**  
검증 로직과 폼 입력 흐름을 한 방식으로만 테스트하면 실패 원인을 찾기 어렵습니다. 연락처, 이메일, URL, 기간 순서 같은 정책은 순수 함수 단위 테스트로 확인하고, 글자 수 카운트, 경력 추가/삭제, preview 표시처럼 사용자 조작이 중요한 흐름은 React Testing Library로 검증했습니다.

## 반응형과 접근성

- 모바일에서는 입력 폼과 미리보기를 전환하며 작성할 수 있도록 구성했습니다.
- 데스크톱에서는 입력 영역과 미리보기 영역을 함께 확인할 수 있도록 배치했습니다.
- 입력 필드는 label로 접근 가능하게 만들고, 오류 상태는 `aria-invalid`, 오류 설명은 `aria-describedby`로 연결했습니다.
- 탭 전환 시 첫 오류 필드 포커스, AlertDialog 닫힘 후 포커스 복귀, `prefers-reduced-motion` 대응을 적용했습니다.
- `openWAX`, `Colour Contrast Analyzer`로 기본 접근성 구조와 색 대비를 점검했습니다.

## 테스트

```bash
npm run test:run
```

순수 함수 단위 테스트(Vitest)와 사용자 흐름 테스트(React Testing Library)를 나누어 작성했습니다.

- 연락처, 이메일, URL 형식 검증
- 선택 섹션에서 일부 입력 시 나머지 필수값 검증
- 종료일이 시작일보다 빠른 경우 검증
- 연락처 자동 하이픈 포맷팅, 글자 수 카운트
- 경력 추가/삭제, 재직 중 체크 시 종료일 비활성화
- 입력한 기본 정보가 preview에 표시되는지 확인
- 탭에 검증 오류 개수 표시, 첫 오류 필드 target 계산

## 폴더 구조

```text
src/
├── components/          # 공통 레이아웃, UI 컴포넌트
├── features/
│   ├── documents/       # 문서 저장, 내보내기, 공통 작성/검증 흐름
│   ├── resume/          # 이력서 타입, 검증, 폼, 미리보기
│   ├── coverLetter/     # 자기소개서 타입, 검증, 폼, 미리보기
│   └── careerSummary/   # 경력기술서 타입, 검증, 폼, 미리보기
├── layout/              # 기본/에디터 레이아웃과 문서 설정
├── pages/               # 라우트 단위 페이지
├── router.tsx           # React Router 설정
└── utils/               # 공통 유틸과 단위 테스트
```

## 배포

Vite 빌드 결과물인 `dist` 폴더를 Cloudflare Workers Assets로 배포합니다. React Router를 사용하는 SPA라서 `wrangler.jsonc`에서 직접 URL 접근과 새로고침을 처리하도록 설정했습니다.

```bash
npm run deploy
```
