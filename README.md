# DocKit — 국문 취업 문서 작성 도구

> 입력하면 제출용 문서 형태로 즉시 확인됩니다.

[데모 보기](https://dockit.jsjweb0.workers.dev/) · [GitHub](https://github.com/jsjweb0/dockit)

DocKit 예시 불러오기와 PDF 저장 흐름

---

## 만든 이유

국문 이력서, 자기소개서, 경력기술서처럼 제출용 취업 문서는 형식이 중요하지만,
Word나 한글 없이 일관된 레이아웃으로 작성하고 출력하기가 번거롭습니다.
HTML 표 마크업과 문서별 입력 폼, 실시간 미리보기를 함께 제공해 이 문제를 해결했습니다.

**개발 기간**: 2026.04 ~ 진행 중 (1인 개발)

---

## 주요 기능

| 기능            | 설명                                                                                        |
| --------------- | ------------------------------------------------------------------------------------------- |
| 실시간 미리보기 | 입력값이 즉시 제출용 문서 형태로 반영됩니다.                                                |
| 자동 저장       | 작성 중 이탈 방지 안내와 localStorage 저장으로 작업 손실을 줄였습니다.                      |
| PDF 저장        | 브라우저 인쇄와 `@media print` 스타일을 사용해 텍스트 선택 가능한 PDF로 저장할 수 있습니다. |
| 입력값 검증     | 문서별 필수값, 형식 오류, 기간 순서 오류를 입력 흐름 안에서 확인할 수 있습니다.             |
| 최근 작성 목록  | 여러 문서를 저장하고 이어 작성할 수 있도록 문서 메타 정보를 관리합니다.                     |

현재 국문 이력서, 자기소개서, 경력기술서 작성 흐름을 동일한 문서 작성 구조로 제공합니다.

---

## 기술 스택과 선택 이유

- **React 19 + TypeScript**: 섹션 반복 추가/삭제, 실시간 미리보기 동기화에 컴포넌트 모델이 적합
- **Context API (no 상태관리 라이브러리)**: 문서별 에디터 상태를 라우트 내부 Provider로 제한해 Redux/Zustand 없이 관리
- **Radix UI**: AlertDialog, Tabs, Tooltip 등 키보드 접근성이 중요한 UI를 안정적으로 구성
- **Vite + Cloudflare Workers**: 빌드 속도와 무료 정적 배포 + SPA 라우팅 지원
- **localStorage**: 백엔드 없이 MVP 검증. `meta.version` 필드로 추후 스키마 마이그레이션 대비

## 설계 결정

### 양식 확장을 고려한 구조

DocKit은 새로운 양식을 추가할 때 수정 범위를 최소화하는 구조에 중점을 두었습니다.
반복되는 UI는 공통 컴포넌트로 분리하고, 양식별 제목, 입력 항목, 안내 문구 등 변경되는 정보는 데이터로 관리했습니다. 문서 라우팅과 에디터 설정은 `editor.config.ts`에서 관리해, 새 양식이 필요할 때 Provider, 샘플 데이터, 최근 문서 저장소를 설정으로 연결할 수 있도록 했습니다.

### 검증 로직을 직접 구현한 이유

DocKit은 입력값, 실시간 미리보기, 자동 저장이 같은 문서 데이터를 기준으로 동작해야 했기 때문에 React state로 폼 데이터를 직접 관리했습니다. 검증 로직은 UI 코드와 분리한 순수 함수로 작성해, 화면 렌더링과 관계없이 Vitest 단위 테스트로 확인할 수 있도록 했습니다.

### 문서별 검증 구조 분리

이력서, 자기소개서, 경력기술서는 에러 구조가 서로 다릅니다. 예를 들어 이력서는 기본 정보와 반복 섹션 에러를 나누어 관리하고, 자기소개서는 문항별 에러를 관리하며, 경력기술서는 경력 항목별 필드 에러를 관리합니다.
이 차이를 Provider 안에 직접 구현하면 새 양식을 추가할 때 touched field 관리, 전체 검증, 첫 오류 메시지 계산을 반복해야 했습니다. 그래서 공통 `useDocumentValidation` 훅은 검증 상태와 touched field 흐름만 담당하고, 문서별 `validationAdapter`가 필드 key 생성, 단일 필드 검증, 전체 검증, 에러 개수 계산을 맡도록 분리했습니다.

### PDF 저장 방식

canvas 기반 이미지 캡처는 텍스트 선택이 불가능합니다.
브라우저 인쇄(`window.print`) + `@media print` 스타일로 텍스트 선택 가능한 PDF를 구현했습니다.

## 해결한 문제

**상태 구조 설계**: 각 문서를 하나의 타입(`Resume`, `CoverLetter`, `CareerSummary`)으로 관리해
폼과 미리보기가 같은 데이터를 참조하도록 구성했습니다.

**검증 정책**: "하나라도 입력되었으면 나머지 필수값을 검증"이라는 정책을 구현해
빈 항목을 남겨둔 상태에서 저장하는 오탐을 방지했습니다.

**검증 확장성**: 문서마다 다른 에러 구조는 유지하되, 공통 훅이 바라보는 검증 결과를 `isValid`, `errors`, `firstMessage` 형태로 통일했습니다. 덕분에 저장/출력 전 검증 흐름은 재사용하고, 새 문서는 adapter만 추가해 연결할 수 있습니다.

**접근성**: `aria-invalid`, `aria-describedby`, 탭 전환 시 첫 오류 필드 자동 포커스,
AlertDialog 닫힘 후 포커스 복귀, `prefers-reduced-motion` 대응을 적용했습니다.

## 폴더 구조

```text
src/
├── components/          # 공통 레이아웃, UI 컴포넌트
├── features/
│   ├── documents/       # 문서 저장, 내보내기, 공통 작성 레이아웃/검증 훅
│   ├── resume/          # 이력서 타입, 검증 adapter, 폼, 미리보기
│   ├── coverLetter/     # 자기소개서 타입, 검증 adapter, 폼, 미리보기
│   └── careerSummary/   # 경력기술서 타입, 검증 adapter, 폼, 미리보기
├── layout/              # 기본/에디터 레이아웃
├── pages/               # 라우트 단위 페이지
├── router.tsx           # React Router 설정
└── utils/               # 공통 유틸과 단위 테스트
```

## 테스트

```bash
npm run test:run
```

순수 함수 단위 테스트 (Vitest):

- `validateBasics`: 이름·연락처·이메일 필수값/형식 검증 (공백, 형식 오류 포함)
- `validateOptionalSections`: "일부 입력 시 나머지 필수값 검증" 정책 케이스
- `formatRelativeTime`: 초/분/시간/일 단위 경계값

UI 상호작용 테스트 (React Testing Library):

- `BasicsSection`: 연락처 입력 시 자동 하이픈 포맷 적용 확인
- `BasicsSection`: 간단 소개 입력 시 글자 수 카운터 표시 확인
- `BasicsSection`: 이름·이메일 등 주요 입력 필드가 접근 가능한 라벨로 렌더링되는지 확인

## 배포

이 프로젝트는 Vite 빌드 결과물인 `dist` 폴더를 Cloudflare Workers Assets로 배포합니다.
React Router를 사용하는 SPA이므로 `wrangler.jsonc`에서 직접 URL 접근과 새로고침을 처리하도록 설정했습니다.

```bash
npm run deploy
```

`npm run deploy`는 내부적으로 `npm run build`를 실행한 뒤 `wrangler deploy`로 배포합니다.

## 배운 점

- 한국식 이력서처럼 표 형태가 중요한 화면은 컴포넌트 분리뿐 아니라 인쇄 스타일, 셀 너비, 긴 텍스트 줄바꿈까지 함께 설계해야 완성도가 올라간다는 것을 확인했습니다.
- 실시간 미리보기 화면에서는 폼 상태와 출력 상태가 분리되면 버그를 추적하기 어려워, 하나의 문서 타입을 기준으로 폼과 미리보기를 함께 관리하는 구조가 중요했습니다.
- 선택 입력 섹션은 단순 필수값 검증보다 "일부 입력 시 나머지 항목을 검증"하는 정책이 실제 사용자 입력 흐름에 더 적합했습니다.
- 문서별 에러 구조를 억지로 하나로 맞추기보다, 공통 훅이 사용하는 검증 결과 형태만 통일하고 세부 검증 규칙은 adapter로 분리하는 편이 확장성과 가독성의 균형이 좋았습니다.
- 순수 함수 테스트와 UI 상호작용 테스트의 역할을 나누어, 검증 로직은 독립적으로 확인하고 폼 입력 흐름은 사용자 관점에서 확인했습니다.
